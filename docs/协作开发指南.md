## 1. 前置准备

### 1.1 协作者首次加入项目

```bash
# 克隆仓库（将 YOUR_USERNAME 替换为协作者 GitHub 用户名）
git clone https://github.com/yunjianzhishui/anki-cli.git
cd anki-cli

# 查看远程仓库
git remote -v
# 应显示 origin 指向 https://github.com/yunjianzhishui/anki-cli.git
```

### 1.2 每次开发前：同步最新代码

```bash
cd anki-cli

# 拉取远程最新 master
git fetch origin
git checkout master
git pull origin master
```

## 2. 创建新分支

每次开发新功能或修复问题时，从最新的 `master` 创建新分支。

### 2.1 基本流程

```bash
# 1. 确保在 master 且已是最新
git checkout master
git pull origin master

# 2. 创建并切换到新分支
git checkout -b <分支名>

# 示例：开发「导出 CSV 支持自定义分隔符」功能
git checkout -b feature/csv-delimiter

# 示例：修复「复习时卡片重复显示」的 bug
git checkout -b fix/review-duplicate-card

# 示例：更新文档
git checkout -b docs/update-readme
```

### 2.2 分支命名建议

| 类型 | 前缀 | 示例 |
|------|------|------|
| 新功能 | `feature/` | `feature/export-csv` |
| Bug 修复 | `fix/` | `fix/sync-timeout` |
| 文档 | `docs/` | `docs/api-reference` |
| 重构 | `refactor/` | `refactor/cmd-structure` |
| 依赖/配置 | `chore/` | `chore/upgrade-anki` |

## 3. 在分支上协作开发

### 3.1 本地修改与提交

```bash
# 查看当前修改
git status

# 添加指定文件
git add ankicli/main.py
git add docs/协作开发指南.md

# 或添加所有修改
git add .

# 提交（建议写清 commit message）
git commit -m "feat: 支持 CSV 导出时自定义分隔符"

# 多次修改可多次提交
git add .
git commit -m "fix: 处理空字段时的边界情况"
```

### 3.2 提交信息规范（可选）

```
<类型>: <简短描述>

<类型> 示例：
  feat: 新功能
  fix: 修复 bug
  docs: 文档
  refactor: 重构
  test: 测试
  chore: 构建/依赖/配置
```

### 3.3 协作者之间的同步

若多人同时在同一分支协作：

```bash
# 推送前先拉取他人提交
git pull origin <当前分支名>

# 若有冲突，解决后：
git add .
git commit -m "merge: 解决与 xxx 的冲突"
git push origin <当前分支名>
```

## 4. 推送更新到远程

### 4.1 首次推送新分支

```bash
# 推送并设置上游分支
git push -u origin <分支名>

# 示例
git push -u origin feature/csv-delimiter
```

### 4.2 后续推送同一分支

```bash
# 已设置过 -u 后，直接 push 即可
git push

# 或显式指定
git push origin feature/csv-delimiter
```

### 4.3 推送被拒绝时（远程有更新）

```bash
# 先拉取并变基
git pull --rebase origin <分支名>

# 解决冲突后
git add .
git rebase --continue

# 再推送
git push origin <分支名>
```

## 5. 合并到 master

### 5.1 方式一：通过 GitHub Pull Request（推荐）

1. **创建 PR**
   - 打开 https://github.com/yunjianzhishui/anki-cli
   - 点击 **Pull requests** → **New pull request**
   - **base** 选 `master`，**compare** 选你的分支（如 `feature/csv-delimiter`）
   - 填写标题和描述，点击 **Create pull request**

2. **代码审查**
   - 协作者可在 PR 中评论、建议修改
   - 作者根据反馈继续在分支上提交，推送后 PR 自动更新

3. **合并**
   - 审查通过后，点击 **Merge pull request**
   - 选择合并方式：
     - **Create a merge commit**：保留完整合并历史
     - **Squash and merge**：将分支多次提交压成一次（历史更简洁）
     - **Rebase and merge**：线性历史，无合并提交

4. **删除远程分支（可选）**
   - 合并后 GitHub 会提示删除分支，可点击删除

### 5.2 方式二：本地合并后推送

```bash
# 1. 切回 master 并更新
git checkout master
git pull origin master

# 2. 合并分支
git merge feature/csv-delimiter

# 若有冲突，解决后：
# git add .
# git commit -m "merge: 合并 feature/csv-delimiter"

# 3. 推送到远程
git push origin master

# 4. 删除本地分支（可选）
git branch -d feature/csv-delimiter

# 5. 若远程分支已无用，可删除
git push origin --delete feature/csv-delimiter
```

## 6. 继续迭代更新

合并到 master 后，下一轮开发从最新的 master 重新开始。

### 6.1 开始新一轮开发

```bash
# 1. 同步最新 master
git checkout master
git pull origin master

# 2. 创建新分支
git checkout -b feature/next-feature

# 3. 开发、提交、推送...
```

### 6.2 长期分支的同步（如 develop）

若使用 `develop` 等长期分支：

```bash
# 在 develop 上
git checkout develop
git pull origin develop

# 将 master 的更新合并进来
git merge master

# 解决冲突后推送
git push origin develop
```

### 6.3 分支已落后 master 很多时

```bash
# 在功能分支上
git checkout feature/old-feature

# 将 master 的更新变基到当前分支
git fetch origin
git rebase origin/master

# 若有冲突，解决后：
# git add .
# git rebase --continue

# 强制推送（因为 rebase 改写了历史）
git push --force-with-lease origin feature/old-feature
```

## 7. 常见场景速查

| 场景 | 命令 |
|------|------|
| 查看所有分支 | `git branch -a` |
| 查看当前分支 | `git branch` 或 `git status` |
| 切换分支 | `git checkout <分支名>` |
| 放弃本地未提交修改 | `git checkout .` 或 `git restore .` |
| 查看某分支与 master 的差异 | `git diff master..<分支名>` |
| 拉取并合并远程分支 | `git pull origin <分支名>` |
| 删除本地分支 | `git branch -d <分支名>` |
| 删除远程分支 | `git push origin --delete <分支名>` |
| 暂存当前修改（切换分支前） | `git stash` |
| 恢复暂存 | `git stash pop` |

## 8. 推荐分支命名规范

```
feature/<功能简述>     # 新功能，如 feature/export-csv
fix/<问题简述>         # Bug 修复，如 fix/sync-error
docs/<文档范围>        # 文档，如 docs/readme
refactor/<模块>       # 重构，如 refactor/review-module
chore/<事项>          # 杂项，如 chore/deps-update
```
